## Findings

### P2 — Non-blocking

**#1 Dual export of `initWasm`/`setWasmSource`**

- **Design doc**: §2.4 — `wasm-source.ts` wraps `wasm.ts` for worker-safe use
- **Code**: [`index.ts`](file://packages/docx-core/src/index.ts#L10-L22) re-exports from both `wasm.ts` and `wasm-source.ts`
- `wasm-source.ts` wraps `wasm.ts`'s `initWasm`/`setWasmSource` with worker-source tracking. Both versions land in the public API, producing duplicate entries in `dist/index.d.ts`. Consumers may be confused about which to use.
- **Recommendation**: Document in `index.ts` that `wasm-source.ts` exports are the preferred entry point for applications that use workers; the `wasm.ts` direct exports are the low-level API.

**#2 `fflate` is a direct dependency not imported by source**

- **Design doc**: §4 Phase 1 step 8 — explicitly requires adding `fflate`
- **Code**: [`package.json`](file://packages/docx-core/package.json#L29) lists `fflate: ^0.8.2`
- `fflate` is not imported by any source file in `docx-core/src/`. It is a transitive dependency of `fast-png` (confirmed in upstream `pnpm-lock.yaml`). Adding it as a direct dependency is defensive and matches the task spec, but it's a dead dependency at the direct level.
- **Recommendation**: Retain as-is per task spec. If `fast-png` ever drops `fflate` as a dependency, this explicit pin prevents breakage.

### P3 — Informational

**#3 `declare const Buffer` for Node.js compatibility**

- **Code**: [`image-render.ts:5`](file://packages/docx-core/src/image-render.ts#L5) — `declare const Buffer: { ... }`
- From upstream. Used for base64 encode/decode in Node.js runtimes. Harmless in browser contexts (declared but never called unless TIFF processing is triggered in Node). May trigger linter warnings in strict DOM-only type setups.
