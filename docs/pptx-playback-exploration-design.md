# PPTX 播放探索版设计

## 1. 文档状态

- 状态：探索已完成，作为历史证据保留
- 日期：2026-07-11
- 目标：先通过真实代码和真实渲染探索 PPTX 播放能力，再决定正式产品设计
- 完成日期：2026-07-11
- 代码性质：试验代码，不进入公开包，不代表正式支持

本文记录已经完成的三段播放探索、测试方法和实际结论，不再指导正式开发顺序。当前实现入口是 [PPTX 播放正式实现设计](pptx-playback-implementation-design.md)，具体任务以 [PPTX 播放工程化路线图](pptx-playback-roadmap.md) 为准。探索代码和结果继续保留，用来说明正式设计为何采用对象身份、时间节点树和属性轨道。

## 2. 核心判断

这个项目的未知部分不是普通工程问题，而是以下问题能不能真正跑通：

- PowerPoint 的时间节点能否还原成浏览器可执行的播放顺序；
- 动画目标能否稳定对应到静态渲染后的对象；
- 浏览器动画能否和当前渲染器的位置、旋转、缩放共存；
- 文字、图表、SmartArt、媒体、页面切换和复杂触发器的实际难度有多大；
- `pptx-vue-viewer` 的解析与播放代码有多少可以直接借鉴；
- `@aiden0z/pptx-renderer` 只需小补丁，还是需要较大的维护分支。

这些问题不能靠文档分析得出最终答案。探索版应尽快形成一条真实播放链路，并尽可能广地尝试 PPTX 的主要播放机制。代码可以临时、重复和不完整，但每个结论必须来自可重复运行的 PPTX。

## 3. 探索版不追求什么

本轮明确不做：

- 不设计 `@arcships/pptx-core` 的正式播放接口；
- 不修改 `@arcships/vue-pptx` 的公开组件；
- 不接入 Worker；
- 不做服务器端渲染；
- 不做完整错误分类和资源限制；
- 不做性能优化；
- 不做正式包构建和工作区外消费；
- 不要求 Chromium、Firefox、WebKit 同时工作；
- 不建立完整截图差异门槛；
- 不为试验代码补齐全面单元测试；
- 不承诺探索成功的效果一定进入正式版本。

默认只使用 Chromium。某项能力明显依赖浏览器实现时，再用 Firefox 或 WebKit 做一次对照。

## 4. 探索版最终要交付什么

探索完成时应有以下结果：

1. 一个可以选择真实 PPTX 并执行播放的试验应用；
2. 一批覆盖主要播放机制的 PPTX 素材；
3. 每类能力的实际运行录屏或截图；
4. 一张结果表，标明可行、部分可行、错误、阻断和未尝试；
5. 对象映射、时间解析、动画执行和页面切换的试验代码；
6. `pptx-vue-viewer` 可借鉴部分和不可直接复用部分的代码结论；
7. 当前静态渲染器需要修改的位置和补丁规模；
8. 基于实际结果提出的正式支持范围、架构建议和工作量估计。

探索版的价值不是代码本身，而是让后续设计建立在已经跑过的结果上。

## 6. 试验应用界面

页面只需要服务调试，不做正式产品外观。

### 6.1 主区域

- 选择本地 PPTX；
- 页面列表和当前页；
- 播放舞台；
- 上一页、下一页；
- 下一动画、上一动画；
- 播放本页、暂停、继续、重置；
- 跳到指定页面；
- 全屏；
- 播放速度：0.25、0.5、1、2 倍。

### 6.2 调试区域

至少显示：

- 当前幻灯片原始 `p:transition`；
- 当前幻灯片原始 `p:timing`；
- 解析后的时间树；
- 当前单击批次；
- 正在运行和已经结束的动画；
- 动画目标 `spid` 和对应页面元素；
- 当前对象的静态位置与动画变化；
- 未识别 XML 节点；
- 浏览器动画和媒体错误；
- 可复制的事件记录。

页面提供“显示目标边框”开关。打开后，每个识别到的动画对象显示形状编号，方便肉眼确认是否找对对象。

## 7. 打开文件与静态渲染

当前静态预览调用 `PptxViewer.open()`，原始 `PptxFiles` 被封装在查看器内部。探索版直接使用第三方库公开入口：

```ts
const files = await parseZipLazyMedia(buffer, RECOMMENDED_ZIP_LIMITS)
const presentation = buildPresentation(files, { lazySlides: true })
viewer.load(presentation)
```

试验应用同时保存：

- 原始 `ArrayBuffer`；
- `PptxFiles`；
- `PresentationData`；
- 当前 `SlideData`；
- 当前页面原始 XML；
- 当前页面关系文件和媒体关系；
- 当前页面渲染出的页面元素。

浏览器主线程直接使用 `DOMParser` 读取 XML。本轮不为 Worker、传输对象和安全解析做额外封装。

## 8. 对象映射试验

PPTX 动画通过 `p:spTgt/@spid` 指向形状。当前渲染器已经把 `cNvPr/@id` 读取到对象模型，但没有写入渲染后的页面元素。

探索版直接修改固定版本的渲染器，让每个对象元素带上：

```html
data-pptx-node-id="42"
data-pptx-node-type="shape"
```

本轮允许采用任一最快方式：

1. 使用 `pnpm patch` 修改已安装版本；
2. 克隆固定提交并用 Vite 别名指向本地源码；
3. 临时复制渲染入口到试验应用。

先选择修改量最小的方式，不在本轮讨论长期维护。需要实际测试：

- 顶层形状；
- 图片；
- 组合对象和组合内子对象；
- 表格；
- 图表；
- SmartArt；
- 来自版式或母版的对象；
- 同一页重复编号或找不到编号的异常情况。

页面元素顺序可以作为临时调试信息，但不能作为最终判断。探索报告要明确记录哪些对象能稳定对应，哪些需要进一步修改渲染器。

## 9. 时间解析试验

解析器不先追求漂亮的数据类型。第一步完整保留 XML 节点、属性和父子关系，再逐步提取实际执行需要的字段。

优先读取：

- `p:timing`；
- `p:tnLst`；
- `p:seq`；
- `p:par`；
- `p:cTn`；
- `p:stCondLst`；
- `p:endCondLst`；
- `p:cond`；
- `p:childTnLst`；
- `p:subTnLst`；
- `p:anim`、`p:animEffect`、`p:animMotion`、`p:animRot`、`p:animScale`、`p:set`、`p:cmd`；
- `p:audio`、`p:video`；
- `p:bldLst`；
- `p:extLst` 以及 p14、p15 扩展。

解析结果同时提供两种视图：

- 原始时间树：尽量接近 XML，用于理解结构；
- 临时执行列表：按单击批次整理，用于快速驱动动画。

如果临时执行列表无法表示某个真实文件，优先记录结构并针对该文件修改，不提前建立复杂通用模型。

## 10. `pptx-vue-viewer` 的使用方式

`pptx-vue-viewer` 不作为第二个静态渲染器接入产品。探索阶段重点研究以下代码：

- 原生动画 XML 解析；
- 动画预设编号和类别映射；
- 单击批次整理；
- 对象触发和悬停触发；
- 文字逐段展开；
- 播放状态推进；
- React 播放层如何使用解析结果；
- Vue 播放层为何没有完整接入原生动画结果。

允许把少量相关文件复制到试验应用中快速修改，但必须保留原项目许可说明、原始文件路径和固定提交号。探索报告分别记录：

- 可以直接借鉴；
- 需要适配当前静态模型；
- 只能作为思路参考；
- 实际运行不正确。

不要复制它的完整页面模型和编辑器，也不要为了复用动画代码替换当前静态渲染底座。

## 11. 动画执行方式

探索版优先使用浏览器动画接口。每个效果实现为一个普通函数：输入目标页面元素、解析参数和播放时间，返回浏览器动画对象或临时控制句柄。

允许：

- 直接修改 `opacity`、`visibility`、`clip-path`、`filter`；
- 给对象临时增加动画外层；
- 使用独立的 `translate`、`rotate`、`scale`；
- 对无法准确还原的预设写一个近似效果，用于判断方向是否可行；
- 对完全不理解的效果只记录信息，不执行。

每个近似效果必须在调试面板标记为“近似”，不能和按原参数执行混在一起。

同一对象多段动画发生冲突时，本轮先保留浏览器实际结果和冲突记录，不急于设计正式的属性合并系统。

## 12. 页面切换试验

页面切换使用前后两张页面叠放。探索以下内容：

- 无效果和直接切换；
- 淡化；
- 推进、覆盖和擦除；
- 分割、随机条、棋盘和形状遮罩；
- 自动换页；
- 切换声音；
- Morph；
- 三维切换。

Morph 和三维切换不要求完成。需要通过实际代码回答：

- 能否从文件中可靠识别；
- 浏览器是否可以用现有对象模型近似；
- 近似结果是否有产品价值；
- 如果要接近 PowerPoint，需要补哪些跨页对象和图形信息。

## 13. 播放特性测试矩阵

探索版按机制覆盖，不按动画名称逐个完成。每类至少准备一个单项文件和一个组合文件。

| 类别 | 必须尝试的内容 | 主要观察 |
|---|---|---|
| 开始方式 | 单击时、与上一动画同时、上一动画之后 | 单击分组和并行关系 |
| 时间设置 | 延迟、持续时间、重复、自动反向、保持、恢复 | 开始结束状态和计时 |

| 进入 | 出现、淡入、飞入、擦除、缩放、旋转 | 初始隐藏和方向参数 |
| 强调 | 透明度、颜色、放大缩小、旋转、闪烁 | 与静态样式是否冲突 |
| 退出 | 消失、淡出、飞出、缩放 | 最终状态和重播恢复 |
| 运动路径 | 直线、折线、曲线、自定义路径 | 坐标和静态位置叠加 |
| 目标对象 | 形状、图片、组合子对象、表格、图表、SmartArt | `spid` 是否能落到正确元素 |
| 文字 | 整体、按段落、按词、按字、项目符号 | 是否需要拆分现有文字 DOM |

| 页面切换 | 淡化、推进、擦除、遮罩、Morph、三维 | 双页叠放与跨页对象能力 |
| 对象触发 | 单击形状、悬停、超链接、内部跳转 | 事件命中和普通下一步的关系 |
| 自动播放 | 自动换页、循环演示、排练计时 | 页内动画结束与换页计时 |
| 媒体 | 音频、视频、自动播放、循环、书签触发 | 关系解析和浏览器限制 |
| 控制 | 暂停、继续、重播、后退、跳页、全屏 | 状态能否恢复或重建 |
| 复杂组合 | 同一对象多效果、多对象并行、动画中切页 | 冲突、取消和残留状态 |

## 14. 素材准备

### 14.1 单项文件

使用桌面版 PowerPoint 制作。每个文件尽量只包含一种机制，文件名直接表达内容，例如：

```text
timing-on-click.pptx
timing-with-after.pptx
entrance-fade-fly.pptx
motion-custom-path.pptx
text-by-paragraph.pptx
chart-by-series.pptx
trigger-shape-click.pptx
transition-morph.pptx
media-bookmark.pptx
```

### 14.2 组合文件

准备 3—5 个专门用于制造复杂情况的文件，每个文件包含多页、多个对象和不同触发方式。

### 14.3 真实文件

收集 5—10 个实际业务 PPTX。探索版不要求正式授权登记，但不能把敏感业务文件提交到仓库。敏感文件只记录哈希、页数和观察结果。

### 14.4 PowerPoint 结果

每个单项文件保存一段 PowerPoint 播放录屏。探索阶段主要肉眼比较以下内容：

- 动画作用对象；
- 单击次数；
- 大致顺序；
- 大致方向；
- 初始和最终状态；
- 页面切换是否明显不同。

本轮不做像素级差异和严格计时误差。

## 15. 结果记录

`results/feature-matrix.md` 每一行记录：

| 字段 | 内容 |
|---|---|
| 功能 | 被尝试的播放机制 |
| 素材 | 文件和页面 |
| PowerPoint | 原本表现 |
| 探索版 | 实际表现 |
| 状态 | 可行、部分可行、错误、阻断、未尝试 |
| 偏差 | 对象、顺序、时间、画面或交互问题 |
| 原因 | 当前判断或未知 |
| 改动 | 修改过的解析器或渲染器位置 |
| 证据 | 截图、录屏、事件记录 |
| 下一步 | 继续、换方案、正式实现或放弃 |

状态定义：

- **可行**：主要行为已经跑通，正式实现问题明确；
- **部分可行**：可以播放，但结果或模型明显不完整；
- **错误**：执行了，但对象、顺序或画面不正确；
- **阻断**：现有静态模型或浏览器能力不足，无法继续；
- **未尝试**：尚未进入代码验证。

探索阶段不使用“已支持”。“可行”只表示技术方向可行。

## 16. 实施顺序

### 第 1 段：打通完整链路

状态：已于 2026-07-11 完成探索验证。

目标是在最短时间内看到来自原 PPTX 的真实对象动画：

1. 建立试验应用；
2. 显式得到 `PptxFiles`；
3. 显示原始 `p:timing`；
4. 给渲染对象增加形状编号；
5. 解析 `seq`、`par`、三种开始方式、延迟和持续时间；
6. 执行出现、淡入、消失、淡出；
7. 实现下一动画、重置和下一页；
8. 完成一次淡化页面切换。

实际结果：真实 PPTX 已在 Chromium 中按多个单击批次、同时开始和上一动画之后运行；出现、淡入、消失、淡出以及统一淡化切页均已执行。详细证据见 `apps/pptx-playback-lab/results/`。

### 第 2 段：快速扩展覆盖面

状态：已于 2026-07-11 完成全部八类探索验证。

按测试矩阵依次尝试：

1. 强调和简单移动；
2. 曲线运动路径；
3. 文字逐段；
4. 对象点击触发；
5. 更多页面切换；
6. 音频和视频；
7. 自动换页和暂停继续；
8. 图表与 SmartArt 内部动画。

每项以尽快得到“能不能、为什么”为目标。发现阻断后先记录，立即转到下一类，不在一个难点上持续工程化。

实际结果：文字逐段、对象点击、视频播放与暂停、自动换页、暂停继续已经跑通；强调、曲线路径、推进与擦除切换、音频和 SmartArt 得到可运行的近似结果；图表内部动画确认被当前静态渲染模型阻断。音频资源读取和播放已经跑通，但声音节点与嵌套时间树的精确对应尚未完成。详细证据见 `apps/pptx-playback-lab/results/feature-matrix.md`。

### 第 3 段：挑战高难能力并收口

状态：已于 2026-07-11 完成全部六类探索验证，探索阶段收口。

集中尝试：

- Morph；
- 三维页面切换；
- 媒体书签；
- 复杂嵌套时间树；
- 同一对象的并行动画冲突；
- 真实业务文件。

然后整理结果表、补丁规模、代码借鉴结论和正式方案建议。

实际结果：Morph 可以在顶层对象中做部分匹配和位置尺寸补间，但缺少可靠的跨页对象身份；旋涡三维切换可以做空间感近似，不能还原原算法；媒体书签可以按时间命中并触发动画；深度 6 的主序列、交互序列和自动往返可以运行；同对象并行动画确认必须进行属性级冲突裁决；43.4 MB、79 页真实业务文件的首屏和媒体交互均已跑通。详细结果与正式实现建议见 `apps/pptx-playback-lab/results/`。

正式实现前又完成了三个定向代码验证：显式名称和创建编号可以作为跨页强身份；最小时间节点树完整覆盖三组现有效果和执行顺序；属性轨道消除了淡入淡出的矛盾最终状态，并能同时保持位移、缩放和强调。正式设计可以据此固定为稳定对象身份、时间节点树和属性轨道三部分。

## 17. 什么时候结束探索

不是等全部效果都正确才结束。满足以下条件即可结束：

1. 测试矩阵的每个大类都至少实际尝试过一次；
2. 基础播放链路可以在多个真实文件中运行；
3. 高难能力已经知道主要阻断位置；
4. `pptx-vue-viewer` 可复用范围有代码结论；
5. 当前渲染器的修改规模已经可以判断；
6. 能基于结果提出可信的正式版本范围和架构。

如果某个大类无法完成，也要保留失败代码、文件和原因。失败结果同样是探索产物。

## 18. 探索结束后的决策

探索报告根据实际结果选择后续方向：

| 结果 | 后续方向 |
|---|---|
| 时间解析和对象映射都可行 | 重新设计正式播放核心和公开接口 |
| 时间解析可行，对象映射需要小补丁 | 向渲染器提交改动，同时保留短期补丁 |
| 时间解析可行，渲染器需要大改 | 评估维护分支或替换静态底座 |
| 基础动画可行，高级效果成本高 | 发布明确的分级支持范围 |
| 只有近似播放有价值 | 产品明确选择“近似播放”，不宣称原效果还原 |
| 基础时间顺序也无法稳定还原 | 停止正式播放器路线，保留静态预览 |

正式的包边界、Worker、错误合同、性能、三浏览器验证和发布门槛，都在这个决策之后重新设计。
