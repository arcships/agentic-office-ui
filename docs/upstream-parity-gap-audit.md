# Upstream Parity Gap Audit — DOCX / XLSX

Date: 2026-07-06

## Purpose

This document records the strict source-level audit requested after the earlier visual-acceptance pass was challenged. It is not an acceptance record. It is the local source of truth for currently known unimplemented or incomplete upstream-parity work in the Vue port.

Conclusion: the current Vue DOCX/XLSX implementation is not 1:1 with the upstream React packages. It contains useful scoped viewer/editor behavior, but many upstream capabilities are missing, simplified, or exposed as API shells/no-ops/placeholders.

## DOCX gaps

### DOCX summary

The local Vue DOCX implementation is a scoped/basic port, not a full React DOCX equivalent. Evidence:

- Upstream README advertises read-only viewer, richer editor/controller API, pagination, page layout, theme, tracked-change, form-field, thumbnail hooks, configurable page surfaces, night-reader mode, and lower-level OOXML/model/layout/serializer exports: `/Users/eric8810/Code/extend-ui-upstream/react-docx/packages/react-viewer/README.md:5`.
- Upstream playground imports and uses `useDocxFormFields`, `useDocxImageWrapMenu`-related APIs, `useDocxPageThumbnails`, paragraph styles, line spacing, borders, track changes, comments, context menus, form dialogs, thumbnails, zoom, read-only toggles, image insertion, and table actions: `/Users/eric8810/Code/extend-ui-upstream/react-docx/apps/playground/src/App.tsx:17`.
- Local Vue `useDocxEditor` exposes a smaller controller and has multiple no-ops: `setFontSize`, `setFontFamily`, `setTextColor`, `setHighlight`, `setSuperscript`, `setSubscript`, `setLineSpacing`, `setParagraphStyle`, table operations, `insertImage`, `applyBorderPreset`, form-field methods, comments/tracked changes arrays, paragraph style/line-spacing selectors, and thumbnails: `packages/vue-docx/src/composables.ts:192`.
- Local `docx-core` WASM bridge remains partially stubbed: `packages/docx-core/src/wasm.ts:87`, `packages/docx-core/src/wasm.ts:169`, `packages/docx-core/src/wasm.ts:179`.

## XLSX gaps

- Upstream README advertises workbook previews, provider/controller API, frozen panes, tables, merged cells, conditional formatting, sparklines, selection, resizing, copy/paste, zoom, embedded images/shapes/form controls/charts/chartsheets, worker-backed parsing, guardrails, and thumbnails: `/Users/eric8810/Code/extend-ui-upstream/react-xlsx/packages/react-xlsx/README.md:5`.

- Local controller declares a similar interface but has many no-ops: `defineNamedRange`, `fillSelection`, `mergeSelection`, chart/image move/resize/update, `recalculate`, row/column resize, paste, formula/style/range style setters, selected style, chart/image rects, `unmergeSelection`, and thumbnail paint: `packages/vue-xlsx/src/composables.ts:930`.
- Local `download()` writes JSON summary with `.xlsx` extension and `exportXlsx()` delegates to it: `packages/vue-xlsx/src/composables.ts:932`, `packages/vue-xlsx/src/composables.ts:969`.

## Cross-cutting acceptance blockers

1. **No-op APIs must not be treated as parity.** Any exposed method returning empty arrays, null, false, or doing nothing is a gap unless explicitly documented unsupported and hidden/disabled in UI.
2. **Export/download must be real.** DOCX/XLSX controls that produce invalid files or JSON summaries with Office extensions are blockers.
3. **Demo pages must not overclaim.** Page titles, descriptions, API tables, and evidence must use scoped/partial language until the gap list is closed or intentionally re-scoped with agreement.
4. **Visual acceptance cannot close source-level parity gaps.** Browser smoke checks prove stability, not upstream parity.
5. **Engine decisions are prerequisite for full parity.** DOCX WASM/serializer and XLSX workbook/worker/export engines must be completed before claiming full upstream equivalence.

## Required follow-up process

1. Keep this document updated whenever a gap is implemented, intentionally re-scoped, or newly discovered.
2. For each High or Medium gap, add implementation evidence: files changed, tests, browser route/viewport, fixture, and export/reimport proof where applicable.
3. Update `docs/visual-acceptance-handoff.md` when this gap list changes because that document governs final acceptance.
4. Do not request final six-page visual acceptance until all High/Medium gaps are either implemented and verified or explicitly marked as intentional gaps with user/project agreement.
