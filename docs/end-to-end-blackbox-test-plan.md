## 2. 黑盒边界

黑盒测试只能通过以下入口观察系统：

- 五个包在发布压缩包中的公开导出；
- 正式构建后的 demo 页面和用户可见控件；
- 浏览器公开事件、DOM、Canvas、iframe、下载和文件选择行为；
- 浏览器控制台、页面异常、网络请求和响应头；
- 导出文件能否被独立工具重新打开；
- 进程、Worker、对象 URL、缓存和内存能否在操作后释放。

以下方式不能作为黑盒通过证据：

- 直接调用内部函数并断言返回值；
- 只确认某个按钮、方法名或 DOM 节点存在；
- 只运行 `tsc`、构建或结构测试；
- Worker 失败后静默退回主线程，但测试仍记为 Worker 通过；
- 页面写着 `READY`、`PASSED` 或 `implemented`；
- 用工作区源码别名代替真实发布包消费；
- 失败后删除断言、跳过用例或放宽为“未抛异常”。

核心单元测试、Vue 组件测试仍然需要建设，但它们与本文的黑盒检查相互补充，不能互相替代。

## 3. 当前可运行入口与目标入口

### 3.1 当前基础命令

```bash
pnpm install --frozen-lockfile
pnpm typecheck
pnpm build
node packages/vue-docx/tests/verify-integration.mjs
node packages/vue-xlsx/test/structure.mjs
pnpm audit --prod --audit-level moderate
pnpm --filter demo preview --host 127.0.0.1 --port 4173
```

这些基础命令有已知局限：

- `pnpm typecheck` 已覆盖 Vue 单文件组件，但仍不等于运行时验证；
- XLSX 结构测试读取构建后的 `dist`，只检查空控制器的公共方法；
- DOCX 集成测试允许 Worker 回退主线程；
- `pnpm build` 成功不代表浏览器资源地址和发布压缩包可用。

因此，执行者必须如实记录这些结果，但不能据此宣布黑盒通过。

### 3.2 P1 统一命令

P1-CI-01 已建立以下根级真实脚本：

```text
pnpm check
pnpm test:unit
pnpm test:component
pnpm test:blackbox
pnpm test:consumer
pnpm test:docs
pnpm test:release
```

每个脚本必须失败时返回非零退出码，并把证据写到固定目录。

当前 P1 套件内容如下：

- `test:unit` 运行 DOCX/XLSX 核心行为、既有 DOCX 集成、XLSX 结构和材料清单检查；
- `test:component` 运行三个 Vue 包的状态、事件、卸载、对象 URL 和 Worker 生命周期检查；
- `test:blackbox` 只使用正式构建，依次运行路由直达/刷新、DOCX/XLSX 业务流程和六项竞态回归；
- `test:consumer` 重新打五个真实 tgz，在工作区外安装，执行类型、正式构建和浏览器 Worker/WASM 检查；
- `test:docs` 检查本文、执行手册、根命令和实际脚本是否一致；
- `test:release` 当前运行 P1 根门禁，不代表候选版本的 `BB-RELEASE`；候选发布仍按 P4 和全新会话执行。

路线图兼容别名 `test:e2e` 与 `test:pack` 分别指向 `test:blackbox` 与 `test:consumer`，不得形成两套不同实现。

### 3.3 自动化测试入口约定

P0/P1 整改必须同时提供一个只使用公开接口的测试消费页。它不是新的产品 API，而是正式构建测试入口，至少提供以下稳定元素：

| 入口 | 约定 |
|---|---|
| 页面状态 | `[data-testid="page-status"]`，`data-state` 只能是 `idle/loading/ready/error` |
| 当前材料 | `[data-testid="loaded-file"]`，显示当前文件名 |
| DOCX 来源 | `[data-testid="docx-import-source"]`，显示 `worker` 或 `main-thread` |
| DOCX 页面 | `[data-testid="docx-page"]`，带从 1 开始的页码 |
| XLSX 工作表 | `[data-testid="xlsx-sheet-tab"]`，选中项带 `aria-selected` |
| 错误 | `[data-testid="load-error"]`，包含稳定错误码和用户可读信息 |
| 诊断 | 页面通过公开 `onDiagnostic` 事件或等价公开回调报告 requestId、Worker 生命周期、WASM URL、取消和资源释放；不得读取模块私有变量 |

测试消费页还必须允许通过公开配置传入 `wasmUrl`、`workerUrl`、资源上限、是否允许主线程回退和 URL 规则。`DOCX-002`、`DOCX-003`、资源低阈值用例都通过该入口执行。

这些入口由路线图 `P0-DOCX-01`、`P1-E2E-01` 和 `P1-CONSOLE-01` 落地。在它们尚未存在时，相应用例应明确记为 `BLOCKED`，不能由 Agent 自行读取内部状态或把人工猜测记为通过。

`P1-FIXTURE-01` 已生成机器可读的 `test-data/manifest.json`。每项记录文件名、类型、分类、合成来源、预期行为、字节数和 SHA-256，例如：

```json

  "sha256": "<实际值>",
  "type": "docx",
  "category": "normal",
  "source": "synthetic-v1",
  "expected": "preserve invoice text and table structure through roundtrip",
  "bytes": 36004
}
```

所有材料由 `scripts/generate_test_materials.py` 以固定种子、固定时间戳和稳定 ZIP/PDF 元数据生成。`node scripts/ci/verify-fixtures.mjs` 检查完整文件集合、类型签名、分类覆盖、别名和校验值；损坏材料只记录稳定错误类别，不记录随浏览器变化的完整堆栈。

### 5.2 严重级别

| 级别 | 定义 | 发布处理 |
|---|---|---|
| P0 | 页面崩溃、公开包不可安装、脚本执行、数据严重错误、核心 Worker/WASM 完全不可用 | 立即暂停，必须修复 |
| P1 | 核心查看、编辑、导出、恢复或安全限制不可用 | 必须修复 |
| P2 | 明显视觉、键盘、响应式、性能或兼容问题 | 修复；确需保留时必须由负责人书面接受 |
| P3 | 不影响完成任务的文案或细节问题 | 可进入后续计划，但要有负责人 |

安全问题不得以“仅 demo”降级。公开页面、示例和组件库都可能处理不可信内容。

## 8. `BB-SEC-URL`：URL、下载与内容隔离

所有安全用例只能在隔离的 localhost 环境执行。测试脚本只设置本地内存标记，不访问外部攻击地址，不执行系统命令。

| 用例 ID | 优先级 | 输入/操作 | 可观察断言 | 证据 |
|---|---:|---|---|---|
| SEC-001 | P0 | 分别传入 `http`、`https`、`blob`、`javascript`、`data`、`file` 和自定义协议 | 只有明确允许的协议通过；拒绝结果为结构化错误 | 协议矩阵和错误码 |
| SEC-002 | P0 | XLSX `src` 使用只修改 `window.__blackboxMarker` 的 `javascript:` 地址并点下载 | 标记不变化，下载不触发，页面给出不安全 URL 错误 | 标记前后值、console |

## 10. `BB-PACK-CONSUMER`：真实发布包消费

当前人工执行的打包顺序：

```bash
pnpm install --frozen-lockfile
pnpm build
PACK_DIR="$(mktemp -d)/packs"
mkdir -p "$PACK_DIR"
for pkg in packages/docx-core packages/xlsx-core packages/vue-docx packages/vue-xlsx packages/vue-extend; do
  (cd "$pkg" && npm pack --json --pack-destination "$PACK_DIR")
done
find "$PACK_DIR" -maxdepth 1 -name '*.tgz' -print
```

## 12. `BB-STRESS`：安全限制和资源释放

### 12.1 安全生成原则

不得为测试制造会真实耗尽开发机内存或磁盘的文件。PDF 只验证整份文件体积上限，不测试页数、单页像素、总内存、并发页或人为构造的复杂度上限。优先使用以下方法：

- 在测试配置中把 DOCX/XLSX 的输入、解压、XML、图片像素和页数上限降到很小；PDF 则通过公开 `maxFileSize` 把整份文件上限调到略低于现有有效样本；
- 使用几 KB 到几 MB 的可控 ZIP，声明或实际内容刚好越过测试上限；
- 使用尺寸元数据超限但内容很小的图片头；
- Worker 超时使用人为延迟，不使用无限循环；
- 所有恶意 URL 指向 localhost，只修改内存标记；
- 测试结束必须删除临时文件和停止服务器。

## 15. 视觉、无障碍和跨浏览器规则

### 15.1 视觉回归

- UI 外壳、工具栏、表格头、错误态和空态使用稳定截图比较；
- 文档、Canvas、光标和动画区域单独截图或遮罩动态部分；
- 首次建立基线必须人工确认，不能拿当前错误页面自动设为基线；
- 截图差异超过批准阈值时保存差异图并人工判断；
- 修复逻辑问题后必须重建相关截图，不能只更新快照。

### 15.2 无障碍

至少检查：语义名称、角色、禁用/选中/展开状态、Tab 顺序、焦点可见、Escape 关闭、颜色对比、缩放 200%、输入错误提示和 Canvas 的可替代说明。自动扫描只能发现部分问题，键盘路径必须真实操作。

### 15.3 跨浏览器

若某能力依赖浏览器原生 PDF、剪贴板、文件系统或下载差异，必须在接口文档中说明，并为每个浏览器写出可观察断言。不能因为 Chromium 通过就默认其他浏览器通过。

## 16. 失败、重试与已知差异

1. 单个失败最多自动重试一次，首次失败证据必须保留；
2. 重试后通过仍标记 `flaky`，不能直接记为稳定通过；
3. 同一用例在干净环境稳定失败两次，即建立缺陷，不继续无限重试；
4. 产品缺陷与测试环境缺陷分开记录；没有证据时按产品失败处理；
5. 已知差异必须包含影响、负责人、接受人、到期时间和对应用例；
6. P0、安全 P1、数据错误、公开包不可用和静默主线程回退不得列为已知可接受差异；
7. 黑盒执行 Agent 发现产品问题后只记录，不在同一验收运行中修改产品代码。

## 17. 自动化建设顺序

当前文档先保证任何 Agent 可以按手册人工驱动；随后按以下顺序落地自动脚本：

1. 六路由、控制台、网络和正式资源冒烟；
2. DOCX Worker/WASM 与 Editor 启动；
3. npm 压缩包独立消费；
4. URL/PDF/CSV 安全用例；
5. 快速切换和取消；
6. DOCX/XLSX 编辑与导出再导入；
7. PDF 和 Components 完整交互；
8. 性能、压力、视觉和跨浏览器。

自动化实现应放在明确的测试目录，并提供根脚本。实现前，路径和命令只属于计划，不得写成“已存在”。

## 18. 最终通过条件

候选版本只有同时满足以下条件才可发布：

- `BB-P0-ROUTES`、`BB-DOCX-WORKER`、`BB-SEC-URL`、`BB-RACE`、`BB-PACK-CONSUMER`、`BB-PERF-XLSX`、`BB-STRESS`、`BB-RELEASE` 全部有完整证据；
- 所有 P0/P1 已关闭，未接受的 P2 为零；
- 六个正式路由在三种浏览器和四个视口完成核心路径；
- 真实 Worker/WASM、真实发布压缩包和导出再导入得到证明；
- 控制台、页面异常和关键网络失败清零；
- 安全限制在拒绝后能恢复正常文件；
- 候选 tgz 校验值与被测试文件一致；
- 新 Agent 只读当前核心文档可以重复执行并得到一致结论；
- 项目负责人确认所有明确保留的 P2 差异。

任何一项仅有源码说明、demo 文案或手工口头确认而没有证据时，视为未通过。
